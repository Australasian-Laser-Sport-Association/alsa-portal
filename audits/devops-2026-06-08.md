## Findings

### [High] CRON_SECRET compared with a non-timing-safe `===` (addressed 2026-06-08)
**Location:** `api/admin/event.js:29-33`
**Evidence:**
```js
function isCronRequest(req) {
  const expected = process.env.CRON_SECRET
  if (!expected) return false
  return req.headers.authorization === `Bearer ${expected}`
}
```
**Impact:** String `===` short-circuits on the first differing byte, leaking timing
about the secret; the correct primitive is a constant-time compare. The endpoint is
the backup-run cron path, which on success emails the full registrations/payments/
events dataset to the DB-configured committee recipients — so a forged match
triggers an off-schedule backup but cannot redirect recipients. Bounded impact, but
it violates the secret-handling invariant.
**Recommendation:** (deferred — see remediation note above; replaced with
`crypto.timingSafeEqual` on length-guarded Buffers)
**Verification needed:** None — pattern confirmed in source.

### [High] `react-router-dom ^7.13.1` — two HIGH npm-audit advisories
**Location:** `package.json:22`; `npm audit`
**Evidence:**
```
react-router  7.0.0 - 7.14.2   Severity: high
  - turbo-stream unauthenticated RCE (GHSA-49rj-9fvp-4h2h)
  - open redirect via protocol-relative URL (GHSA-2j2x-hqr9-3h42)
  - XSS in unstable RSC redirect (GHSA-8646-j5j9-6r62)
  - stored XSS via unescaped Location header (GHSA-f22v-gfqf-p8f3)
  - DoS via __manifest endpoint (GHSA-8x6r-g9mw-2r78)
  - DoS via reflected single-fetch input (GHSA-rxv8-25v2-qmq8)
2 high severity vulnerabilities
```
**Impact:** This is a client-only SPA (`createBrowserRouter`, no React Router SSR),
so the RCE (turbo-stream), Location-header, manifest, and single-fetch advisories
target the server/SSR data layer and are largely not reachable in this deployment
shape. The open-redirect and client-routing XSS items can apply. Real risk, partly
mitigated by architecture.
**Recommendation:** (deferred)
**Verification needed:** Confirm the patched version range and that no SSR/RSC data
APIs are introduced later.

### [Medium] Rate limiting exists only on the contact form
**Location:** rate limiting present only in `api/contact.js:15-39` (`@upstash/ratelimit`, 3/24h per IP)
**Evidence:** `@upstash`/`Ratelimit` appears in `api/contact.js` only. No limiting on
`api/referee-test.js`, `api/volunteer-signup.js`, `api/player.js`, `api/captain.js`,
`api/profiles.js`, `api/admin/*`, or `api/superadmin/[resource].js`. The manual
backup-run path (`api/admin/event.js?resource=backup-run`, committee auth) emails
via Resend with no throttle.
**Impact:** The only public unauthenticated POST (contact) is correctly limited, so
blast radius is moderate; but authenticated-but-abusable endpoints (repeated
referee-test submissions, `profile-search` typeahead scans, committee-triggered
backup emails) have no abuse ceiling — the backup-email trigger is an
email-amplification vector.
**Recommendation:** (deferred)
**Verification needed:** None.

### [Low] Source maps may ship publicly if `SENTRY_AUTH_TOKEN` is unset
**Location:** `vite.config.js:59,67`
**Evidence:**
```js
build: { sourcemap: true }
sentryVitePlugin({ sourcemaps: { filesToDeleteAfterUpload: ["./dist/**/*.map"] } })
```
**Impact:** The `.map` files are deleted from `dist` only when the Sentry upload step
runs (requires `SENTRY_AUTH_TOKEN` in the Vercel build env). If the token is absent,
the plugin no-ops the upload and the source maps may ship to production publicly.
**Recommendation:** (deferred)
**Verification needed:** Whether `SENTRY_AUTH_TOKEN` is set in the Vercel build env
(operational).
