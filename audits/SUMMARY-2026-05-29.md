# ALSA Portal — Security Review Summary (Pass 1)

**Prepared for:** ALSA Committee
**Date:** 2026-05-29
**Status of this review:** Pass 1 of 6 complete (security & authorization). Passes 2–6 outstanding.

---

## Executive summary

The ALSA Portal underwent a structured security review covering access control, authentication, the payment system, and the public-facing surface. The review combined an automated code analysis with manual verification against the live database, so findings reflect how the system actually behaves in production rather than assumptions from the code alone.

The review found **four critical issues, all of which have now been fixed and verified.** Each concerned access control — situations where a member could potentially read or change data they should not, such as another member's payment details, their own outstanding fees, or administrative permissions. None required sophisticated hacking; they were the kind of thing a curious or motivated member could attempt from an ordinary browser. All four are closed, confirmed by live testing.

A number of lower-priority issues were also identified and are tracked for remediation before and shortly after launch. Importantly, the live verification showed that the portal's core access-control model is fundamentally sound: the large majority of the database is correctly locked down, and several issues that looked alarming on paper turned out to be already safe in practice.

**Bottom line:** the portal is in good security shape. The critical issues are resolved, the remaining items are understood and planned, and the fact that a rigorous review found and closed these before the wider rollout is a mark of diligence, not a red flag. This is a normal and healthy outcome for a thorough pre-launch audit.

---

## What was reviewed and how

Scope (Pass 1): Supabase row-level security policies, authorization logic, authentication and sessions, the payment/reconciliation system, the public website surface, serverless functions, and third-party dependencies.

Method: a static code review produced an initial findings list, which was then verified against the production database directly — checking actual table permissions, security policies, and exploitability — rather than relying on the code or the documentation. Several findings were confirmed, several were downgraded once the live behaviour was checked, and one new issue was discovered during verification.

Dependencies were scanned and returned **zero known vulnerabilities**. The server-side secret key was confirmed to be server-only and absent from the public application.

## The four critical issues (all resolved)

1. **Administrative privilege escalation.** A committee member could grant themselves or anyone else the highest "superadmin" role, or lock out an existing superadmin. This existed both in an admin function and in a database permission. *Fixed:* role changes now require superadmin authority at both layers, and members cannot edit their own permissions through that path.

2. **Claiming another member's record.** A logged-in member could absorb another member's "placeholder" record — including any registrations or recorded payments attached to it — simply by being signed in. *Fixed:* the system now verifies the record genuinely belongs to the claimer (matching alias or email) before allowing it.

3. **Exposure of event bank details.** Event banking details (BSB, account number, account name) were readable by anonymous, logged-out visitors directly from the data layer — a classic ingredient for payment-redirection scams. *Fixed:* anonymous access to those fields has been removed; registered members can still see where to pay.

4. **Members altering their own registration data.** A registered member could change their own outstanding fee to zero, mark themselves as having met eligibility requirements (including the under-18 approval and referee-test gates), or confirm their own status, by editing their registration directly. *Fixed:* a database safeguard now blocks members from changing these protected fields while leaving the legitimate self-service actions (side-event and dinner selections) working normally.

## Remaining work (planned, not blocking)

These are real and should be addressed before and around launch, but none is a "anyone can do it right now" critical:

- **Payment ledger integrity.** Recorded payments can currently be edited or deleted by any committee member with no change-history. Recommend making the ledger append-only or adding an audit trail. (Insider-integrity, not external exploit.)
- **Event deletion.** Any committee member can permanently delete an entire event and its registrations/payments without a superadmin gate or confirmation. Recommend a superadmin gate plus a typed confirmation.
- **Pre-nationals fee self-setting.** The newer competition registration path lets a member set their own fee on signup, and its fee-lock has a bug. Recommend the same fix already applied to the main registration table.
- **Member directory enumeration.** A logged-in member can list other members' roles and details in bulk. Recommend constraining this to teammates, self, and committee.
- **Defense-in-depth hardening.** ~12 tables grant broader write permission than their security policies actually use. The policies correctly block misuse today, but tightening the permissions removes the risk of a future change accidentally opening a hole. Recommend one consolidated hardening change.

Lower-priority items (password strength floor, security headers, rate limiting on logged-in actions, account-deletion path, public-roster name exposure, side-event partner consent) are catalogued in the technical working record for scheduling.

## What this review did not cover

Pass 1 was security and authorization only. A complete picture requires the remaining passes, each run the same rigorous way: database integrity (Pass 2), code quality and maintainability (Pass 3), performance (Pass 4), accessibility and user-experience correctness (Pass 5), and operations, monitoring, and documentation (Pass 6 — including adding error tracking so issues surface automatically rather than via member reports).

---

## Recommendation

The portal is safe for the committee's review and continued rollout. We recommend: completing the High-priority remediation list before opening registrations more widely; scheduling the defense-in-depth hardening; and running the remaining audit passes to confirm the rest of the system meets the same standard the security layer now does.
