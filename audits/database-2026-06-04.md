### [High] Cascade deletes erase the payment ledger AND bypass the audit trail
**Location:** `supabase/migrations/20260514000000_payment_tracking.sql:85`,
`20260527060000_competition_payment_records_integration.sql:36-47`,
`20260424000000_event_sweep.sql:25-27`, `20260604010000_payment_records_history.sql`
**Evidence:**
```sql
-- payment_records FKs, both ON DELETE CASCADE:
registration_id             uuid REFERENCES public.zltac_registrations(id)        ON DELETE CASCADE
competition_registration_id uuid REFERENCES public.competition_registrations(id)  ON DELETE CASCADE

-- zltac_registrations cascades from BOTH its parents:
user_id uuid NOT NULL REFERENCES public.profiles(id)        ON DELETE CASCADE   -- initial_schema:181
ADD CONSTRAINT zltac_registrations_year_fk
  FOREIGN KEY (year) REFERENCES public.zltac_events(year)   ON DELETE CASCADE   -- event_sweep:25

-- competition_registrations cascades from competitions:
competition_id uuid NOT NULL REFERENCES public.competitions(id) ON DELETE CASCADE -- pre_nats:98
```
The audit table is written *only* by the two SECURITY DEFINER RPCs
(`edit_payment_record`, `delete_payment_record`); there is **no `AFTER DELETE`
trigger on `payment_records`** (confirmed: the only delete-side trigger in the
whole migration set is `on_auth_user_deleted` on `auth.users`).
**Impact:** The `payment_records_history` table was built with no FK precisely so
a delete can't erase the audit trail — but that protection only covers deletes
that go through the `delete_payment_record` RPC. Every *cascade* delete skips the
RPC and leaves **no history row at all**. Four real paths trigger a cascade that
silently destroys payment ledger rows with zero audit:
1. Deleting a `zltac_events` row (or its `year`) → cascades to all that year's
   `zltac_registrations` → cascades to all their `payment_records`. (The event
   `delete` API path was flagged in Pass 1; here the *schema itself* would wipe
   the ledger even without the API loop.)
2. Deleting a `competitions` row → cascades to `competition_registrations` →
   `payment_records`.
3. Deleting a user (`auth.users` → `cleanup_profile_on_auth_delete` deletes the
   `profiles` row → cascades to `zltac_registrations` → `payment_records`).
4. Deleting an individual `zltac_registrations` / `competition_registrations` row.
This is the inverse of the stated design goal. The ledger is the
"highest business-risk surface"; the audit table gives a false sense of
completeness because it is empty for exactly the most destructive operations.
**Recommendation:** Add an `AFTER DELETE` trigger on `payment_records` (SECURITY
DEFINER, `search_path = public`) that snapshots `OLD` into
`payment_records_history` with `operation = 'delete'` and a sentinel
`changed_by` (e.g. NULL = "cascade/system"), so cascades are captured the same
way RPC deletes are. This is strictly additive to the existing RPC path (the RPC
deletes will then produce two history rows unless the RPC's manual snapshot is
removed — switch the RPC to rely on the trigger, or guard the trigger to skip
RPC-originated deletes). Separately, reconsider whether `ON DELETE CASCADE` is
the right action from `zltac_registrations`/`competition_registrations` to
`payment_records` at all — `RESTRICT` would force an explicit, audited teardown
before any registration carrying payments can be removed.
**Verification needed:** Confirm live state matches the files:
```sql
-- ON DELETE actions on the two payment_records FKs (a=no action, r=restrict,
-- c=cascade, n=set null):
SELECT conname, confdeltype
FROM pg_constraint
WHERE conrelid = 'public.payment_records'::regclass AND contype = 'f';

-- Confirm there is NO delete-side trigger on payment_records:
SELECT tgname, tgtype FROM pg_trigger
WHERE tgrelid = 'public.payment_records'::regclass AND NOT tgisinternal;
```

### [High] Cents-vs-dollars split across the money tables has no DB-level guard
**Location:** `supabase/migrations/20260527060000_competition_payment_records_integration.sql:8-21`,
`20260526010000_pre_nationals_phase1.sql:103-104`,
`20260514000000_payment_tracking.sql:16,86`,
`api/_lib/computeCompetitionAmountPaid.js:7-9,48-49`
**Evidence:**
```sql
-- payment_records.amount: integer cents, ALWAYS, regardless of which FK is set
amount integer NOT NULL CHECK (amount <> 0)

-- ZLTAC parent: cents
ALTER TABLE public.zltac_registrations ADD COLUMN amount_owing integer NOT NULL DEFAULT 0;

-- Competition parent: DOLLARS, numeric(8,2)
amount_paid  numeric(8,2) NOT NULL DEFAULT 0,
amount_owing numeric(8,2) NOT NULL DEFAULT 0,
```
```js
// computeCompetitionAmountPaid.js — the ONLY place the unit boundary is crossed
const sumCents   = (rows ?? []).reduce((acc, r) => acc + (r.amount ?? 0), 0)
const amountPaid = Math.round(sumCents) / 100          // cents -> dollars
const amountOwing = Number(reg.amount_owing ?? 0)      // already dollars
```
**Impact:** The same column (`payment_records.amount`) means cents in dollars-terms
for competition rows and cents in cents-terms for ZLTAC rows — interpretation
depends entirely on *which FK is non-null*. Nothing in the schema enforces the
conversion; it lives in one JS helper. Any future query that sums
`payment_records.amount` across both arms (a treasurer "total collected" report,
a backup export, a new endpoint) — or any direct SQL the committee runs — will be
off by 100× for one of the two arms and silently produce wrong money totals. The
migration author documented the trap, which is good, but documentation is not a
constraint. The `CHECK (amount <> 0)` is the only check on this column.
**Recommendation:** Pick one canonical unit for `payment_records.amount` (cents is
already the project-wide convention per the initial-schema header) and store
competition parent amounts in cents too, converting at the API/UI edge only —
i.e. migrate `competition_registrations.amount_owing/amount_paid` to integer cents
so all three tables agree. If the dollars representation must stay, add a
generated/companion column or a clearly-named view that normalises both arms to
cents, and route every aggregate through it. At minimum, add a `CHECK` that
competition-arm amounts are whole-cent values (`amount % 1 = 0` after ×100) to
catch fractional-cent drift.
**Verification needed:** Confirm the live column types still match the files
(types are a common out-of-band change):
```sql
SELECT table_name, column_name, data_type, numeric_precision, numeric_scale
FROM information_schema.columns
WHERE table_schema = 'public'
  AND (table_name, column_name) IN
   (('payment_records','amount'),
    ('zltac_registrations','amount_owing'),
    ('competition_registrations','amount_owing'),
    ('competition_registrations','amount_paid'));
```

### [Medium] `doubles_pairs` allows a player to be double-booked; `triples_teams` has no player uniqueness or distinctness at all
**Location:** `supabase/migrations/20260422000000_initial_schema.sql:265-286`
**Evidence:**
```sql
CREATE TABLE public.doubles_pairs (
  ...
  player1_id uuid REFERENCES public.profiles(id) ON DELETE SET NULL,
  player2_id uuid REFERENCES public.profiles(id) ON DELETE SET NULL,
  ...
  UNIQUE (event_year, player1_id),
  UNIQUE (event_year, player2_id)
);

CREATE TABLE public.triples_teams (
  ...
  player1_id uuid REFERENCES public.profiles(id) ON DELETE SET NULL,
  player2_id uuid REFERENCES public.profiles(id) ON DELETE SET NULL,
  player3_id uuid REFERENCES public.profiles(id) ON DELETE SET NULL,
  ... -- NO unique constraints, NO distinctness check
);
```
**Impact:** The two separate uniques on `doubles_pairs` each pin a single column,
so they do **not** prevent player X from being `player1_id` of one pair and
`player2_id` of a *different* pair in the same year — X is entered in two doubles
teams simultaneously. They also don't prevent `player1_id = player2_id` within one
row (a player paired with themselves). `triples_teams` has no uniqueness at all,
so a player can appear in unlimited triples teams for a year, and nothing stops
`player1_id = player2_id = player3_id`. These are the side-event team structures
that drive entries, partner-confirmation, and (via `amount_owing`) money. A
buggy code path or a direct service-role write can create inconsistent rosters
the app's "one entry per player" assumption doesn't expect. (Writes are now
service-role-only after `20260604020000`, which lowers the *exposure* but the
schema still permits the invalid state.)
**Recommendation:** For `doubles_pairs`, the per-player-per-year invariant can't be
expressed with two column uniques; enforce it with an `EXCLUDE`/normalised
approach or a trigger that rejects a player already present in either slot for the
year, plus `CHECK (player1_id IS DISTINCT FROM player2_id)`. For `triples_teams`,
add `CHECK` distinctness across the three slots and an equivalent
"player appears once per year" guard. If full enforcement is too invasive, at
least add the cheap within-row distinctness CHECKs.

### [Medium] No `CHECK (>= 0)` on any money-owing / money-paid column
**Location:** `supabase/migrations/20260514000000_payment_tracking.sql:16`,
`20260526010000_pre_nationals_phase1.sql:103-104`,
`20260422000000_initial_schema.sql:293-301`
**Evidence:**
```sql
amount_owing integer NOT NULL DEFAULT 0          -- zltac_registrations, no CHECK
amount_paid  numeric(8,2) NOT NULL DEFAULT 0     -- competition_registrations, no CHECK
amount_owing numeric(8,2) NOT NULL DEFAULT 0     -- competition_registrations, no CHECK
amount       integer NOT NULL                    -- legacy payments, no CHECK
```
**Impact:** The brief asks directly: can `amount_owing` go negative? At the DB
layer, yes — there is no constraint. A committee/manager edit or a service-role
recompute bug can set `amount_owing` negative (a player "owed" −$50), which the
ZLTAC payment-status logic and any "outstanding balances" report will then
mis-total. `amount_paid` is derived by `computeCompetitionAmountPaid` and could go
negative if refund records out-weigh payments, which the helper maps to `'unpaid'`
rather than surfacing the anomaly. `payment_records.amount` deliberately allows
negatives (refunds), which is correct and should stay — this finding is about the
*parent* aggregate columns only.
**Recommendation:** Add `CHECK (amount_owing >= 0)` to `zltac_registrations` and
`CHECK (amount_owing >= 0 AND amount_paid >= 0)` to `competition_registrations`.
If genuine negative balances (account credit) are a real business state, model
them explicitly rather than letting them arise silently.
**Verification needed:** Before adding the constraint, check no existing row would
violate it:
```sql
SELECT 'zltac' AS src, count(*) FROM public.zltac_registrations WHERE amount_owing < 0
UNION ALL
SELECT 'comp_owing', count(*) FROM public.competition_registrations WHERE amount_owing < 0
UNION ALL
SELECT 'comp_paid',  count(*) FROM public.competition_registrations WHERE amount_paid  < 0;
```

### [Medium] ZLTAC `payment_reference` collision check runs under caller RLS, so the suffix loop never fires for player self-inserts
**Location:** `supabase/migrations/20260522000000_simplify_payment_reference.sql:34-61`,
`20260514000000_payment_tracking.sql:56-73`
**Evidence:**
```sql
CREATE OR REPLACE FUNCTION public.generate_payment_reference(p_year integer, p_alias text, p_id uuid)
RETURNS text LANGUAGE plpgsql VOLATILE AS $$   -- NOT SECURITY DEFINER, no SET search_path
...
  WHILE EXISTS (
    SELECT 1 FROM public.zltac_registrations
    WHERE payment_reference = candidate AND id <> p_id   -- runs under caller RLS
  ) LOOP ...
```
`zltac_registrations` still grants `INSERT` to `authenticated`
(`role_grants_baseline.sql:70`), and the BEFORE INSERT trigger that calls this is
also SECURITY INVOKER. The player's RLS (`zltac_registrations_select_own`) only
admits their own rows.
**Impact:** When two distinct players share an alphanumeric alias and register for
the same year, each player's collision scan sees only their own (zero) rows, both
compute the same `2026CROUCHY`, and the second `INSERT` fails on the
`payment_reference` UNIQUE constraint with a raw 23505 surfaced to the user — the
intended `…2`/`…3` suffix path is never exercised. This is a constraint-vs-trigger
correctness gap: the UNIQUE constraint is doing the collision *detection* that the
trigger was supposed to do gracefully. (Flagged in Pass 1 as a Medium; repeated
here because it is squarely a Pass-2 trigger-correctness issue and still present.
Note the *competition* equivalent is now safe only because competition inserts
are service-role-only post-`20260604020000`, so its trigger runs with full
visibility — the ZLTAC arm is the one still exposed because player inserts remain
client-direct.)
**Recommendation:** Mark `generate_payment_reference` and
`set_zltac_registration_payment_reference` `SECURITY DEFINER SET search_path =
public` (matching `is_committee`, `is_reg_open_for_year`, etc.) so the collision
scan sees all rows. Keep the UNIQUE constraint as the final backstop and handle
23505 gracefully for the genuine concurrent-insert race.

### [Medium] Foreign-key columns without a covering index (notably `teams.event_id`)
**Location:** `supabase/migrations/20260502000001_unified_teams_schema.sql:19-24`
(and several others below)
**Evidence:** FK columns with no btree index serving them:

### [Low] `payment_records_history.changed_by` is `ON DELETE SET NULL` — auditor attribution can be erased
**Location:** `supabase/migrations/20260604010000_payment_records_history.sql:46`
**Evidence:**
```sql
changed_by uuid REFERENCES public.profiles(id) ON DELETE SET NULL,
```
**Impact:** This is an append-only audit table whose whole purpose is "who changed
what, when." The *snapshot* of the old actor (`recorded_by`) is correctly a plain
`uuid` with no FK, so it survives. But `changed_by` — the person who performed the
edit/delete — is an FK that goes NULL if that committee member's profile is later
deleted. The trail then says an edit happened, with no "who." For an audit record,
the actor id should be retained verbatim even after the actor's account is gone.
**Recommendation:** Store `changed_by` as a plain `uuid` (no FK), consistent with
the `recorded_by` snapshot column in the same table; or use `ON DELETE NO ACTION`
/ keep the value. The attribution should outlive the actor's profile.

### [Low] `referee_test_results` has no `UNIQUE (user_id)` despite the API treating it as one-row-per-user

### [Low] `computeCompetitionAmountPaid` classifies status with floating-point equality
**Location:** `api/_lib/computeCompetitionAmountPaid.js:48-55`
**Evidence:**
```js
const amountPaid  = Math.round(sumCents) / 100   // float dollars
const amountOwing = Number(reg.amount_owing ?? 0) // float dollars
...
else if (amountPaid === amountOwing) status = 'paid'
else status = 'overpaid'
```
**Impact:** Comparing two floats with `===` after a `/100` can mis-classify a
fully-paid registration as `'overpaid'` (or `'partial'`) for amounts where the
float representation differs in the last bit. The blast radius is small (status
label only; the stored `amount_paid` is correct), but it is a latent
money-display bug that grows with non-round prices. This is the app-side
counterpart to the unit-drift finding above — it exists *because* the comparison
happens in dollars rather than in integer cents.
**Recommendation:** Compare in integer cents: convert `amount_owing` to cents and
compare `sumCents` to `owingCents` with integer equality, deriving the status
before converting back to dollars for storage.

### [Info] `event_year` columns are plain integers with no FK to `zltac_events.year`
**Location:** `supabase/migrations/20260519040000_legal_event_year_scoping.sql:24-25,46-47`,
`20260519000000_zltac_history_unification.sql:79` (`zltac_event_placings.tournament_year`)
**Evidence:** `legal_acceptances.event_year integer NOT NULL`,
`under_18_approvals.event_year integer NOT NULL`,
`zltac_event_placings.tournament_year integer NOT NULL` — all documented as a
deliberate "plain int" convention rather than FKs.
**Impact:** A compliance acceptance or approval can be recorded against a year that
has no `zltac_events` row (typo, or a year deleted later). For
`legal_acceptances`/`under_18_approvals` this is a compliance ledger, so an
orphaned year is a (low) data-quality risk; for `placings` it is intentional
(historic years predate the events table). Calling out the inconsistency, not
asserting it must change — the convention is internally consistent and the
event-deletion-cascade concern argues *against* adding a CASCADE FK here.
**Recommendation:** Leave `placings` as-is. Consider a validation (app-level or a
deferred FK with `ON DELETE RESTRICT`/`NO ACTION`) for the two compliance tables
so an acceptance can't be written against a non-existent year, without
introducing a cascade that would let an event delete erase compliance records.

## Ruled out (challenged, not findings)

- **xor CHECK constraints are correct.** `chk_teams_event_xor_competition` and
  `chk_payment_records_event_xor_competition` use `(a IS NULL) != (b IS NULL)`,
  which is a true xor over the two nullability states — exactly one FK set. No
  off-by-one. Good.
- **`payment_records_history` no-FK design is correct.** `payment_record_id` is
  deliberately FK-free so the trail outlives a parent delete; RLS is read-only
  committee, with the only writers being the two service-role RPCs. Sound.
  (The gap is that *cascades* don't reach those RPCs — see the High finding —
  not the table design itself.)
- **`ON DELETE RESTRICT` on `legal_acceptances.document_id` and
  `alsa_memberships.period_id` are the right call** — they protect referenced
  audit/membership rows from disappearing. These are positive examples of the
  pattern the brief wants.
- **Trigger service-role guards are correct.** `protect_registration_admin_fields`
  and `protect_competition_registration_fields` both gate on
  `auth.uid() IS NOT NULL AND NOT public.is_committee()`, so service-role writes
  (`auth.uid()` NULL) and committee writes pass — the Pass-1 guard bug class is
  not reintroduced here. (Minor: the competition variant is SECURITY INVOKER with
  no `SET search_path`, but every name it references is schema-qualified, so the
  search-path risk is negligible.)
- **Payment-reference and amount-owing triggers are idempotent.** Both only act
  when the target column `IS NULL` (reference) or `IS NULL/0` (owing) on INSERT,
  so re-running or backfilling won't clobber an explicit value.
- **Single-row config tables are correctly pinned.** `referee_test_settings`,
  `backup_settings` use `CHECK (id = 1)` + seed `ON CONFLICT DO NOTHING`. Sound.

## Coverage gaps

Everything below could not be confirmed in a read-only, file-only pass and should
be verified against the live DB before acting:
