### [Critical] Any committee member can elevate themselves (or anyone) to superadmin via /api/admin/users
**Location:** `api/admin/users.js:26-66`
**Evidence:**
```js
if (req.method === 'PATCH') {
  const { error } = await verifyCommittee(req)
  if (error) return res.status(statusForAuthError(error)).json({ error })
  const body = req.body ?? {}
  let update
  if (body.action === 'reset') {
    const { error: superErr } = await verifySuperAdmin(req)
    ...
  } else if (Array.isArray(body.roles)) {
    update = { roles: body.roles }
    ...
  } else if (typeof body.suspended === 'boolean') {
    update = { suspended: body.suspended }
  }
  ...
  const { error: patchErr } = await supabaseAdmin.from('profiles').update(update).eq('id', id)
```
**Impact:** Every PATCH branch other than `action==='reset'` is gated only by
`verifyCommittee`, which admits the four committee roles (`superadmin`,
`alsa_committee`, `zltac_committee`, `advisor`). Any of those users can call
`PATCH /api/admin/users?id=<anyId>` with `{ roles: ['superadmin'] }`, including
`<anyId> = self`, to grant themselves the most powerful role; or with
`{ roles: ['player'] }` to demote a sitting superadmin; or with
`{ suspended: true }` to lock a superadmin out. The write goes through
`supabaseAdmin` so RLS does not save us. AUDIT_BRIEF.md explicitly names this
("Confirm the known committee-can-suspend-superadmin bug is still present");
confirmed present.
**Recommendation:** Require `verifySuperAdmin` for any change to `roles` and
for any `suspended` change targeting a profile whose current roles include
`superadmin`. Refuse self-edits of `roles` and `suspended` even by superadmin
(self-demotion lockout). Reject any request whose `body.roles` contains
`'superadmin'` unless caller is already superadmin.
**Verification needed:** None — gate is read directly off `body`, no extra
checks downstream.

### [Critical] `claim_placeholder_profile` RPC lets any logged-in user steal another player's placeholder data
**Location:** `supabase/migrations/20260526000000_claim_placeholder.sql:51-123`, `api/player.js:681-695`
**Evidence:**
```sql
CREATE OR REPLACE FUNCTION public.claim_placeholder_profile(
  placeholder_id uuid, real_id uuid
) RETURNS jsonb LANGUAGE plpgsql SECURITY DEFINER ...
BEGIN
  IF auth.uid() != real_id AND NOT public.is_committee() THEN
    RETURN jsonb_build_object('ok', false, 'error', 'not authorised');
  END IF;
  ...
  UPDATE public.zltac_registrations SET user_id = real_id WHERE user_id = placeholder_id;
  UPDATE public.doubles_pairs       SET player1_id = real_id WHERE ...
  ...
  DELETE FROM public.profiles WHERE id = placeholder_id;
END;
$$;
GRANT EXECUTE ON FUNCTION public.claim_placeholder_profile(uuid, uuid) TO authenticated;
```
```js
// api/player.js:685
const { data, error } = await supabaseAdmin.rpc('claim_placeholder_profile', {
  placeholder_id,
  real_id: user.id,
})
```
**Impact:** The function's only authorisation guard is `auth.uid() = real_id`
— i.e. "you are claiming for yourself." It does NOT verify that the placeholder
belongs to the claimer (no alias / email / created_by check). Any authenticated
user can either call the public API (`POST /api/player?resource=claim` with an
arbitrary `placeholder_id`) or invoke the RPC directly from the browser console
(`supabase.rpc('claim_placeholder_profile', { placeholder_id: '<any-uuid>', real_id: <own-uid> })`)
and have ALL of the target placeholder's registrations, doubles pairs, triples
pairs, and legacy `payments` rows reassigned to themselves, then the placeholder
profile deleted. This includes registrations that already have `payment_records`
attached (those follow `registration_id` and are transferred too), so an attacker
can absorb someone else's paid registration. The API layer's `handleClaimable`
(GET /claimable) only suggests matches; the POST handler does not enforce them.
**Recommendation:** Inside the function, require that
`(SELECT alias FROM profiles WHERE id = placeholder_id)` matches the caller's
alias case-insensitively, OR that the placeholder's `placeholder_email` matches
the caller's `auth.users.email`. Committee branch (`is_committee()`) may still
bypass. Additionally, narrow `handleClaim` in `api/player.js` to verify the same
before invoking the RPC.
**Verification needed:** None — function source and grant both checked.

### [Critical] Anonymous PostgREST clients can read bank details from `zltac_events` and `competitions`
**Location:** `supabase/migrations/20260422000001_role_grants_baseline.sql:34`, `supabase/migrations/20260514000001_event_bank_details.sql:6-9`, `supabase/migrations/20260527000000_competitions_anon_discovery.sql:44-68`
**Evidence:**
```sql
-- 20260422000001_role_grants_baseline.sql
GRANT SELECT ON public.zltac_events TO anon;
-- 20260514000001_event_bank_details.sql
ALTER TABLE public.zltac_events
  ADD COLUMN bank_bsb            text,
  ADD COLUMN bank_account_number text,
  ADD COLUMN bank_account_name   text;
-- 20260527000000_competitions_anon_discovery.sql
CREATE POLICY "competitions_discovery_read" ON public.competitions
  FOR SELECT TO anon, authenticated
  USING (archived_at IS NULL AND (registration_close_at IS NULL OR registration_close_at > now()));
GRANT SELECT ON public.competitions TO anon;
```
The migration even acknowledges this explicitly:
```
-- 20260527000000:18-21  "Bank details and other sensitive fields are NOT
--   filtered at the row level. Column-level filtering is handled by the
--   public API endpoint (Phase 3b)."
```
**Impact:** The `/api/public?resource=competitions` and `?resource=event`
handlers do strip the bank columns from their JSON response, but the underlying
table-level `GRANT SELECT TO anon` means anyone with the embedded anon key
(public, see `src/lib/supabase.js`) can hit PostgREST directly:
`GET https://<project>.supabase.co/rest/v1/zltac_events?select=bank_account_name,bank_bsb,bank_account_number&status=in.(open,closed,archived)`
and pull every event's bank account name, BSB, and account number. Same for
`competitions` for any non-archived, not-yet-closed pre-nats event. This is the
class of information used in social-engineering bank-transfer scams ("Hi I'm the
ALSA treasurer, here's the BSB/account: ..."). The migration comment shows the
author is aware and has not yet implemented the column-level mitigation it
recommends.
**Recommendation:** Either (a) revoke `SELECT (bank_account_name, bank_bsb,
bank_account_number)` from anon and authenticated on both tables and grant
column-listed SELECT on every other column (the recipe is spelled out in the
migration's own comment block); or (b) move bank columns to a separate table
gated behind a committee-only RLS policy.
**Verification needed:** Confirm with a live PostgREST request from a logged-out
browser that the bank columns are returned today.

And the only grant in the migration history:

The migrations include this RLS policy that *would* admit the UPDATE if the
GRANT layer allowed it:
```sql
-- 20260522050000_enforce_registration_lock.sql:78
CREATE POLICY "zltac_registrations_update_own" ON public.zltac_registrations
  FOR UPDATE TO authenticated
  USING (user_id = auth.uid())
  WITH CHECK (user_id = auth.uid() AND public.is_reg_open_for_year(year));
```
**Impact:** Two possibilities, both bad:
1. **The UPDATE grant exists in production but is not in migrations** (drift —
   the codebase has precedent: `20260514000002_payment_records_grant.sql` notes
   "Applied to production via the SQL editor as a hotfix"). If so, players can
   directly UPDATE their own `zltac_registrations` row, and the RLS WITH CHECK
   only pins `user_id` and the open-phase gate — it does NOT pin
   `amount_owing`, `payment_reference`, `admin_override_coc`,

history. Strongly recommend revoking direct UPDATE on `zltac_registrations`
from authenticated and routing every confirm/edit through the API
(`/api/player?resource=registration` already exists for cancel/recompute; extend
it for side-events and extras), then dropping the
`zltac_registrations_update_own` RLS policy. RLS WITH CHECK without column-level
locking is not enough for a row that holds the money fields.
**Verification needed:** Run `SELECT has_table_privilege('authenticated',
'public.zltac_registrations', 'UPDATE');` against production. If true, the
fix is urgent.

### [High] Anonymous user metadata is trusted by `handle_new_user` to populate `profiles` columns including `state`, `phone`, `emergency_contact_*`
**Location:** `supabase/migrations/20260428000000_handle_new_user_metadata.sql:9-52`
**Evidence:**
```sql
CREATE OR REPLACE FUNCTION public.handle_new_user()
RETURNS trigger LANGUAGE plpgsql SECURITY DEFINER SET search_path = public AS $$
BEGIN
  BEGIN
    INSERT INTO public.profiles (
      id, first_name, last_name, alias, dob, phone, state, home_arena,
      emergency_contact_name, emergency_contact_phone, roles
    ) VALUES (
      new.id,
      new.raw_user_meta_data->>'first_name',
      ...,
      new.raw_user_meta_data->>'state',
      new.raw_user_meta_data->>'home_arena',
      ...
      ARRAY['player']::text[]
    )
```
**Impact:** `roles` is hard-coded to `['player']` (good), so this is not a
privilege-escalation surface. However, an attacker scripting signups via the
public Supabase auth endpoint can write arbitrary text into `state`, `phone`,
`dob`, `home_arena`, `first_name`, `last_name`, `alias`, `emergency_contact_name`,
and `emergency_contact_phone` — including HTML-looking payloads or other
profile-pollution junk — without ever loading the signup form. Combined with
`/api/profiles` exposing those fields to any authenticated caller (see separate
finding), it means an attacker can plant their own chosen text in the profile
fields that other users see. Mass-signup also creates rows in `profiles` plus
`auth.users` that have to be cleaned up by hand.
**Recommendation:** Either (1) drop the metadata-driven inserts here and have
the signup page write the profile via a server-side API after auth confirmation
(uses validation already implemented elsewhere), or (2) length-cap and
character-validate every text field in the trigger. Add a Supabase rate limit
on the sign-up endpoint at the infrastructure level.
**Verification needed:** None.

### [High] `/api/profiles` lets any logged-in user enumerate every other user's `roles`, name, alias, state, avatar
**Location:** `api/profiles.js:66-97`
**Evidence:**
```js
if (!Array.isArray(ids) || ids.length === 0) return res.json({ profiles: [] })
...

  .from('profiles')
  .select('id, first_name, last_name, alias, state, dob, avatar_url, roles')

```
**Impact:** A logged-in user submits `{ ids: [...] }` and receives the listed

### [High] `payment_records` (the payment audit log) is freely editable and deletable by any committee member with no history
**Location:** `api/admin/event.js:635-673`, `supabase/migrations/20260514000000_payment_tracking.sql:99-111`
**Evidence:**
```js
if (req.method === 'PATCH') {
  const { id, amountCents, datePaid, bankReference, notes } = req.body ?? {}
  ...
  const { error: updErr } = await supabaseAdmin.from('payment_records').update({
    amount: amountCents,
    recorded_at: datePaid || new Date().toISOString(),
    bank_reference: bankReference?.trim() || null,
    notes: notes?.trim() || null,
  }).eq('id', id)
  ...
}
if (req.method === 'DELETE') {
  ...
  const { error: delErr } = await supabaseAdmin.from('payment_records').delete().eq('id', id)
}
```
RLS:
```sql
CREATE POLICY "payment_records_committee_all" ON public.payment_records
  FOR ALL TO authenticated USING (public.is_committee()) WITH CHECK (public.is_committee());
```
**Impact:** Any of the four committee roles can edit the amount, date, bank
reference, and notes on a recorded payment with no history table, no
"who-changed-what-when" trail beyond the original `recorded_by`/`recorded_at`
(both also overwriteable on PATCH). They can also DELETE a payment outright,
removing the ledger entry entirely. This is the project's "highest business
risk surface" per the brief. A rogue or compromised committee account can move
or zero a player's paid amount and the trail evaporates. No `recorded_by` is
written on PATCH, so the audit trail says someone else recorded the new value.
**Recommendation:** Make `payment_records` append-only at the DB layer
(`REVOKE UPDATE, DELETE ON public.payment_records FROM authenticated` plus a
`payment_records_no_update`/`_no_delete` policy returning false). Corrections
should be modelled as a reversing entry plus a fresh entry. If editability
must be kept, add a `payment_records_history` audit table written by an AFTER
UPDATE/DELETE trigger that records the prior row + the caller's uid.
**Verification needed:** None.

### [High] Committee/manager-set `competition_registrations.amount_owing` can be self-set by player on insert
**Location:** `supabase/migrations/20260526010000_pre_nationals_phase1.sql:288-307`, `:405-408`
**Evidence:**
```sql
CREATE POLICY "competition_registrations_self_insert" ON public.competition_registrations
  FOR INSERT TO authenticated WITH CHECK (user_id = auth.uid());
```
```sql
CREATE OR REPLACE FUNCTION public.set_competition_amount_owing()
RETURNS trigger LANGUAGE plpgsql AS $$
BEGIN
  IF NEW.amount_owing IS NULL OR NEW.amount_owing = 0 THEN
    SELECT price_per_player INTO v_price FROM public.competitions WHERE id = NEW.competition_id;
    NEW.amount_owing := coalesce(v_price, 0);
  END IF;
  RETURN NEW;
END; $$;
```
**Impact:** The self-insert WITH CHECK pins only `user_id`, and the BEFORE
INSERT trigger only overrides `amount_owing` when the incoming value is NULL
or 0. A player INSERTing into `competition_registrations` with
`amount_owing: 0.01` (or any non-zero value) bypasses the price lookup and
ends up with whatever owed amount they chose. The UPDATE policy
(`competition_registrations_self_update_nonpayment`) does block subsequent
self-changes to `amount_owing`, but the damage is done at insert.
**Recommendation:** Either (a) change the trigger to always overwrite
`NEW.amount_owing := coalesce(price, 0)` for self-inserts (drop the
"only if NULL or 0" guard), or (b) revoke INSERT on `competition_registrations`
from authenticated and route registration through `api/superadmin/[resource].js`
`handleCompetitionRegistration` (which already exists and pins the value
through the trigger correctly via service role). Option (a) is one-line and
safer.
**Verification needed:** None — insert path and trigger source both checked.

### [High] Captain `team-completions` API endpoint interpolates user-supplied playerIds into a PostgREST `.or()` filter
**Location:** `api/captain.js:246-255`
**Evidence:**
```js
supabaseAdmin
  .from('doubles_pairs')
  .select('player1_id, player2_id, confirmed')
  .eq('event_year', eventYear)
  .or(`player1_id.in.(${playerIds.join(',')}),player2_id.in.(${playerIds.join(',')})`),
supabaseAdmin
  .from('triples_teams')
  .select('player1_id, player2_id, player3_id, confirmed')
  .eq('event_year', eventYear)
  .or(`player1_id.in.(${playerIds.join(',')}),player2_id.in.(${playerIds.join(',')}),player3_id.in.(${playerIds.join(',')})`),
```
**Impact:** `playerIds` arrives from `req.body.playerIds`, an array of strings

malicious id that's not a clean UUID will simply not be in the set and produce
a 403. The exploit window is narrow but the practice (string-interpolating
arbitrary client values into a PostgREST filter string) is unsafe — a future
refactor that removes or weakens the ownership guard would open PostgREST
filter injection.
**Recommendation:** Validate each `playerIds[i]` with a UUID regex before
interpolation, or build the filter as nested `.in('player1_id', playerIds)`
unions instead of a single `.or()` template string.

### [Medium] `payment_reference` collision-detection in trigger runs under caller RLS, defeating the suffix loop for direct player INSERTs
**Location:** `supabase/migrations/20260522000000_simplify_payment_reference.sql:34-61`
**Evidence:**
```sql
CREATE OR REPLACE FUNCTION public.generate_payment_reference(
  p_year integer, p_alias text, p_id uuid
) RETURNS text LANGUAGE plpgsql VOLATILE
AS $$
DECLARE base_ref text := ...; candidate text := base_ref; n integer := 1;
BEGIN
  WHILE EXISTS (
    SELECT 1 FROM public.zltac_registrations WHERE payment_reference = candidate AND id <> p_id
  ) LOOP
    n := n + 1;
    candidate := base_ref || n::text;
  END LOOP;
  RETURN candidate;
END;
$$;
```
**Impact:** The function is `VOLATILE` but **not** `SECURITY DEFINER` and has
no `SET search_path`. The collision-check `SELECT` runs under the caller's
RLS, which for a logged-in player only admits their own
`zltac_registrations` rows. Result: when two distinct players happen to share
an alphanumeric alias and register for the same year, each player's BEFORE
INSERT trigger sees only their own (zero) prior rows, both compute the same
`2026CROUCHY`, the second INSERT fails on the UNIQUE constraint, and the
